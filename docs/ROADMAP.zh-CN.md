# 产品演进路线图

本路线图推动 `llm-gateway` 从成熟的基础设施组件演进为易于采用、
具备策略治理能力的 LLM 运行时网关。路线图按用户可见成果组织，
而不是按 package 或 provider 数量划分。

路线图只描述演进方向，不构成兼容性承诺。只有通过验收条件，
里程碑才算完成；仅合入孤立的后端类型或 API 不算完成交付。

## 产品定位

`llm-gateway` 是面向企业 LLM 流量、策略感知且兼容 OpenAI API 的
运行时网关。它为托管模型和自托管模型提供统一的认证、路由、配额、
治理、计量、审计与可观测性执行点。

它不是 Prompt 编排框架、推理服务器或进程内计费引擎。

## 现有基线

以下功能已经存在，必须发展而不是重建：

- `internal/dispatch` 拥有路由执行、重试、端点回退和
  显式的跨模型回退。
- `internal/selector` 提供加权 / P2C 选择、冷却、inflight 状态，
  以及成功/延迟感知运行时评分。
- `internal/moderation` 提供输入 / 输出 guard 组合和响应 stream decorator。
- `cmd/console` 是独立控制平面，提供 Web UI、Admin / Viewer 角色和写操作审计。
- 数据平面只发出用量事实和指标；聚合与计费仍由下游系统负责。

这些边界在整个路线图中仍然是架构限制。

## 演进原则

1. **采用优先于广度。** 除非为了解决明确的兼容性缺口，否则不再增加 provider；
   优先改善安装体验、能力证明和可运维性。
2. **一个路由引擎。** 扩展`internal/dispatch`左右的策略解析；
   不要创建并行路由执行包。
3. **解释每一次策略决策。** 模型切换、回退、阻断和脱敏都必须产生稳定的
   reason code，供审计与指标使用。
4. **保持热路径数据较小。** 计费聚合和分析保持在外部
   网关。路由可能会消耗专用的紧凑成本配置文件，而不是
   下游发票模型。
5. **流语义是明确的。** 每个输出策略都声明是否
   对于流响应来说是严格的、缓冲的或尽力而为的。
6. **只在授权模型明确时引入多租户。** 新层级必须定义
   `account` 的所有权、隔离和迁移；它绝不能引入
   同一实体的第二个名称。

## 里程碑 0 — 易于采用的产品体验

目标：新的评估者无需先理解内部 package 依赖图，就能理解、运行、
观察项目并完成性能测试。

### M0.1 — 产品叙述

- 围绕产品成果重写两份 README 的首屏。
- 添加一个紧凑的架构图，显示应用程序、网关策略阶段、
  和上游供应商。
- 将逐 package 的数据平面说明移到 Quickstart 之后或架构文档中。
- 显示控制台和路由决策示例。
- 明确区分当前能力与路线图能力。

验收条件：

- 读者在看到第一条安装命令前，就能识别目标用户、待解决问题和差异化价值；
- 没有README声明依赖于未实现的路线图功能；
- 英文和中文首屏内容在结构上保持一致。

### M0.2 — 单命令本地演示

- 添加包含 MySQL、Redis、网关、控制台、
  模拟上游、网关拥有的schema migration和幂等种子。
- 为演示提供稳定、非秘密的开发凭证。
- 提供一个无需手动 SQL 即可达到就绪状态的命令。
- 启动后打印控制台 URL、API 密钥和可复制的请求。
- 为贡献者保留当前仅限基础架构的 Compose 工作流程。

验收条件：

- 干净的结帐达到了成功的聊天请求和可用的控制台
  一项记录在案的命令；
- 第二次运行是幂等的；
- 启动准备是健康检查驱动的，而不是睡眠驱动的；
- CI 执行相同的演示smoke路径。

### M0.3 — 运维仪表板

- 发送可导入的 Grafana 仪表板，而不是面板愿望清单。
- 涵盖请求率、成功/错误类别、p50/p95/p99 延迟、TTFT、端点
  健康/冷却、速率限制拒绝和使用Outbox失败。
- 将 Prometheus 和 Grafana 添加到可选的可观察性演示配置文件中。
- 记录标签基数和网关内成本聚合的缺失。

验收条件：

- 提交到仓库的仪表板无需手动编辑即可导入到记录中
  Grafana版本；
- 面板由演示流量填充；
- PromQL 表达式根据 CI 或测试中的指标名称进行检查。

### M0.4 — 可重复的性能证明

- 使用一种固定加载工具添加非流式和流式加载场景。
- 测量直接模拟上游与网关以隔离网关开销。
- 报告吞吐量、第一个令牌开销、端到端延迟、分配或
  内存、CPU、错误率和活动流。
- 添加客户端缓慢、客户端断开和上游中流故障场景。
- 检查每个已发布结果的线束和环境元数据。

验收条件：

- 其他贡献者可以从记录的命令中复制报告；
- 结果区分网关开销和配置的上行延迟；
- 绩效工作有基线和回归阈值。

## 里程碑 1 — 可解释的策略路由

目标：调用者请求稳定的虚拟模型，而平台选择一个
根据明确的策略允许具体模型和端点。

### M1.1 — 路由策略 ADR 和域契约

- 定义 `virtual model`、`route policy`、`candidate model`、`constraint` 和
  准确地说是`routing decision`。
- 决定账户/项目策略优先级和默认拒绝/默认允许规则。
- 定义稳定的决策原因代码和审核/指标表示。
- 定义现有的兼容性行为
  `X-Gateway-Fallback-Models` 标头。
- 保持端点执行在`internal/dispatch`；模型链解析发生
  在调度收到输入之前。

验收条件：

- ADR包括故障行为、缓存一致性、回滚和迁移；
- 域合约可以无损地代表现有的路由行为；
- 在热路径查询和缓存结构明确之前，不会开始实施。

### M1.2 — 基于规则的虚拟模型

状态：完成。

- 保留虚拟模型策略和候选顺序/权重。
- 将虚拟模型解析为调用者账户下经过验证的模型链
  权限。
- 首先支持确定性约束：区域、模态/能力、最大值
  尝试，以及允许/拒绝模型列表。
- 通过控制台 API 公开 CRUD、验证和试运行评估。
- 记录请求的模型、路由模型、策略版本和原因代码。

验收条件：

- 现有的具体模型请求行为不变；
- 虚拟模型决策在固定输入下是确定性的；
- 无效或取消订阅的候选项永远不会进入派遣；
- 策略更新遵循记录的缓存一致性合同。

### M1.3 — 延迟和成本目标

状态：完成。

- 重用选择器运行时统计数据来观察健康状况和延迟。
- 引入一个紧凑版本的路由成本配置文件，与计费分开
  定价和发票计算。
- 通过探索保障措施添加有限的延迟/成本评分。
- 提供试运行比较和每个决策得分的解释。

验收条件：

- 路由决策可以根据其策略版本和输入重建；
- 丢失/过时的遥测有明确的中立回退；
- 成本感知路由不会将下游计费调用添加到请求路径中；
- 测试涵盖约束冲突和所有候选项不合格的行为。

基于语义/质量的路由被故意推迟到基于规则的路由
具有生产痕迹、评估数据和商定的质量信号。

## 里程碑 2 — 可插拔策略实施

目标：使网关成为供应商中立的策略执行点，而不是一个
内容安全产品。检测逻辑、企业 DLP 和审核 API
在一份稳定的合同下保留可更换的发动机。

### M2.1 — 策略决策合约

状态：完成。

- 将当前仅错误的审核结果演变为明确的操作：
  `allow`、`deny` 和 `redact`。
- 定义稳定的规则 ID、策略版本、原因代码和安全审核元数据。
- 定义全局/账户/项目/API 密钥范围的策略绑定和优先级。
- 保留保护链扩展点并避免耦合策略评估
  到 Gin 或供应商协议。
- 将引擎故障和不受支持的编辑行为定义为故障关闭。

验收条件：

- 策略评估从不记录匹配的秘密或原始敏感内容；
- 每个突变在审计元数据中都是可见的，无需存储敏感信息
  之前/之后的值；
- 现有的审核配置具有记录的兼容性路径。

### M2.2 — 请求执行和变异执行

状态：完成。

- 为支持的请求协议添加结构化文本提取作为适配器，
  不作为发动机合同的一部分。
- 应用引擎提供的突变，而不需要网关自己进行检测。
- 编辑后安全地重建上游请求，同时保留请求信封
  路由字段和协议转换行为。
- 添加策略绑定持久性、控制台 CRUD 和合成模拟
  引擎决策。

验收条件：

- 原始敏感值未到达上游、日志、跟踪或审计；
- 编辑可跨受支持的内容块结构工作；
- 畸形或不受支持的结构有记录的故障打开/故障关闭
  策略。

### M2.3 — 响应执行模式

状态：完成。

- 定义 `strict-buffered` 和 `best-effort-streaming` 输出模式。
- 添加跨 SSE 帧的解码文本累积，以实现尽力扫描。
- 在客户端交付之前制作严格模式缓冲区并记录其延迟/内存
  成本。
- 保持提供商审核和企业 DLP 作为适配器；参考引擎
  是示例，而不是网关拥有的安全平台。

验收条件：

- API 响应和控制台策略配置公开所选保证；
- 测试涵盖跨传输块和 SSE 帧的敏感术语；
- 后第一个字节违规具有显式截断和审核语义。

## 里程碑 3 — 企业身份和层次结构

目标：仅在部署模型之后添加组织/项目自助服务
授权边界已被证明。

### M3.1 — 身份和多租户 ADR

- 确定当前 `account` 是否映射到组织、租户或计费
  记录并记录迁移。
- 定义项目/应用程序所有权、配额继承、模型订阅、
  API 密钥所有权和 BYOK 边界。
- 在添加新表之前定义租户范围的角色。

### M3.2 — OIDC 和范围 RBAC

- 为控制台添加 OIDC 登录，同时保留打破玻璃操作员路径。
- 不仅在存储/存储库操作中强制执行资源范围的授权
  在 UI 路由中。
- 在审计中包括参与者主题、组织/项目范围和策略版本。

验收条件：

- 跨租户访问测试覆盖每一个管理资源；
- 无法通过直接调用Admin API绕过授权；
- 审计确定人类参与者和受影响的资源范围。

## 下一阶段规划门槛

只有真实厂商或生产部署方提出具体的多租户需求，并明确部署模型、资源
归属边界和授权要求后，才会启动里程碑 3。仅为了补齐假设中的企业能力，
不足以成为启动理由。

语义路由或基于质量的路由同样继续延后，直到具备
生产 trace、评测数据和一致认可的质量信号。

在上述条件满足前，后续工作根据生产反馈选择，不代表已经承诺新的
路线图里程碑。

## 路线图工作完成的定义

每个路线图变更必须包括：

- 持久性数据或 API 行为发生变化的兼容性和迁移说明；
- 单元测试加上用户可见路径的集成或smoke断言；
- 策略决策的指标、原因代码和安全审计行为；
- 公共行为的英文和中文用户文档；
- 已发布的性能基线没有出现无法解释的下降。

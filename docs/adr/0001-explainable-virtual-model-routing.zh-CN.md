# 0001. 在调度之前解决虚拟模型策略

* **状态**：已接受
* **日期**：2026-07-14
* **作者**：Zereker

## 上下文

网关当前解析所请求的具体模型和可选的
`X-Gateway-Fallback-Models` 列表在 `internal/middleware/model_service.go` 中。
`internal/dispatch` 然后执行经过验证的模型链并
`internal/selector` 为每个模型选择一个端点。这场分离已经
将目录授权排除在端点调度之外。

Milestone M1引入了稳定的虚拟模型名称，例如`fast-chat`。
可以将一个虚拟名称映射到多个具体模型，但它不能创建一个
第二个执行引擎或让不受信任的请求绕过订阅。每个
决定还需要一个稳定的解释，可以在不记录的情况下进行审计
提示内容或生成无限的指标标签。

当前认证身份有账户但无可信项目。的
路线图为以后的身份/RBAC 决策保留项目层次结构，因此路由
策略不得从请求标头创建项目多租户。

## 定义

- **具体模型**：全局 `model_services` 目录中存在的名称和
  通过调度执行，例如 `gpt-4o`。
- **虚拟模型**：路由策略拥有的面向调用方的稳定名称。它是
  从未发送到上游端点。
- **路由策略**：映射一个虚拟模型的不可变版本规则集
  到有序/加权的具体候选者和确定性约束。
- **候选模型**：评估订阅的具体目录模型，
  允许/拒绝规则、区域、模式和配置的限制。
- **约束**：确定性资格规则。约束删除了
  候选项；它不选择或调用端点。
- **路由决策**：策略解析产生的不可变的解释，
  包括要求的姓名、策略版本、候选项评估、订购
  合格的链、结果和稳定原因代码。

机器可读的合约是`internal/domain/routing_policy.go`。

## 考虑的选项

### 选项 A：将模型优化放在选择器中

- **如何**：扩展选择器候选以包含来自多个模型的端点
  并让其评分者选择模型和端点。
- **优点**：一次得分传球；直接访问端点运行时统计数据。
- **缺点**：将授权和产品策略与端点健康状况混合在一起，使得
  回退语义不明确，并且在 a 中重复目录/订阅检查
  当前仅选择端点的组件。

### 选项B：添加单独的路由执行引擎

- **如何**：虚拟模型路由拥有自己的重试、回退和调用
  循环，仅委派一些请求进行分派。
- **优点**：最大的实施自由度。
- **缺点**：创建两个重试/流实现、两个尝试跟踪，以及
  最终出现不一致的故障行为。

### 选项 C：在分派之前解析可解释的模型链

- **如何**：策略解析器验证请求的名称并生成
  有序的具体模型链加上决策记录。现有调度和
  选择器不改变地执行该链。
- **优点**：保留当前边界，保持决策可重构，并且
  允许策略演变而不触及流执行。
- **缺点**：解析和端点选择是两个阶段；延迟/健康状况
  稍后的优化需要一个小的只读统计投影。

## 决定

我们选择**选项C**。

### 解析和执行边界

模型链解析发生在 `dispatch.Input` 构建之前。它可能会读到
目录、订阅、有效的路由策略和紧凑路由
遥测投影。它返回：

1. 包含所有接受和拒绝的 `domain.ModelRoutingDecision`
   具有稳定原因代码的候选项；和
2. 由决策代表的经过验证的 `[]*domain.ModelService` 链
   合格的候选项。

`internal/dispatch` 仍然是端点加载的唯一所有者，端点
选择、重试、跨模型回退执行、流式传输和尝试
会计。 `internal/selector` 仍然是端点选择器。两者都不是组件
查询路由策略存储。

### 策略范围和优先级

策略范围为 `global`、`account`，保留 `project`。
选择使用最具体的完整策略：项目，然后账户，然后
全球的。策略不会跨范围合并，因为部分合并会产生
决策难以重构和回滚。

在身份验证生成受信任的项目 ID 之前，项目策略处于非活动状态
并且其所有权/RBAC 模型被接受。标头或请求主体字段必须
未确定项目范围。

具体模型请求是默认允许的，以现有目录为准
订阅检查。虚拟模型请求默认拒绝：没有活动的
他们失败的有效策略为 `virtual_model_policy_not_found`。账户
认购始终是一个硬性约束，任何策略都无法扩大。

### 旧版回退标头

对于具体模型请求，`X-Gateway-Fallback-Models` 保留其当前的
有序、去重、最多三行为。每个接受的回退是
记录为 `legacy_fallback_accepted`；缺失或未经授权的候选项可能会
被记录为拒绝评估但从未进入调度。

对于虚拟模型请求，策略是完整的权限。遗产
header被忽略，记录为`legacy_fallback_ignored`；它不能附加一个
模型或扩大策略。这可以防止调用者控制的标头绕过
允许/拒绝约束。

### 确定性约束和尝试

M1.2 最初支持区域、模态/能力、允许/拒绝模型列表以及
最大尝试次数。拒绝战胜允许。明确的允许列表排除模型
不存在于其中。评估目录存在和账户订阅
在候选项进入模型链之前。

策略最大尝试次数是上限，不能替代网关的
操作尝试上限。有效上限是以下的最小正值
两个来源。候选权重仅影响已经存在的排序/选择
符合资格的候选项；它永远不会覆盖约束。

### 失败行为

- 具体目录丢失或订阅拒绝保留现有的 HTTP 行为；
- 未找到/禁用虚拟策略是客户端可见的拒绝；
- 所有不符合资格的候选项均被拒绝，号码为 `no_eligible_candidate`；
- 策略存储不可用会失败关闭
  `routing_policy_unavailable` 和 HTTP 503；
- 格式错误的活动策略失败关闭，值为 `routing_policy_invalid`，
  出现在健康/运营中，并且永远不会陷入不太具体的问题
  策略;
- 具体请求不依赖于路由策略存储并在一个过程中继续
  策略存储中断。

没有被拒绝的候选项、自由格式的策略消息、提示、凭证或原始信息
请求内容包含在指标标签中。

### 审计、跟踪、用量和指标

完整的决策记录是安全的结构化元数据：请求的模型、虚拟的
标志、结果、原因、策略 ID/版本/范围、候选模型 ID/名称、
资格、来源、顺序和有限原因代码。它不包含提示或
秘密。

跟踪和审计可能会附上完整的结构化决策。使用事件记录
请求的型号、实际路由的型号、策略 ID/版本和终端原因。
指标仅使用有界标签，例如 `outcome`、`reason` 和 `scope_kind`；
策略 ID、账户 ID、项目 ID 和模型名称不是指标标签。

### 热路径查询和缓存结构

持久性查询返回一个不可变的有效策略快照，其中包含所有内容
候选项和限制因素；候选行不会被一一加载。的
解析器缓存密钥是 `(account_id, trusted_project_id, requested_model)` 和
缓存的值包含：

- 正向或负向解析；
- 不可变的策略 ID/版本/范围；
- 编制确定性约束并排序候选；
- 加载/过期时间戳。

第一次未命中执行一个有效策略查询。目录和订阅
查找使用现有的有界缓存； M1.2可以批量处理但不能发出
每个端点的查询。正项和负项都有有限的 TTL。一项策略
write 发布带有策略标识和版本的失效；消费者
不得用较旧的事件替换较新的缓存版本。 TTL 仍然是
发布/订阅不可用时的恢复路径。

### 一致性、回滚和迁移

策略版本是不可变的。激活更新会自动更改
有效版本。进行中的请求保留其已解决的快照；新的
请求在失效或 TTL 过期后观察新版本。回滚
将先前的内容重新发布为新的单调递增版本，保留
明确的审计调度。

迁移是累加的：

1.添加策略存储/API/缓存，具体解析保持默认；
2. 仅当存在有效策略时才启用虚拟名称；
3. 将决策元数据添加到跟踪、用量和控制台试运行输出中；
4. 稍后仅使用可信身份和 RBAC 添加项目范围。

回滚会禁用虚拟策略解析。具体要求和遗产
回退标头继续其现有路径；策略表可能保留
未使用，不影响交付。

## 后果

### 积极

- 调用方可以使用稳定的虚拟名称，而无需更改调度或流；
- 每个模型链决策都可以根据策略版本和输入进行重构；
- 授权保持失败关闭状态，并且不能通过请求标头扩大；
- 具体模型兼容性和策略存储故障隔离是明确的；
- 缓存契约是在持久性实现开始之前定义的。

### 负面/权衡

- 策略解析在调度之前添加有界缓存/存储阶段；
- 在身份和 RBAC 提供可信范围之前，项目策略无法交付；
- 非合并范围策略跨级别重复配置；
- 延迟/成本目标需要明确的紧凑遥测投影
  M1.3 而不是直接耦合到选择器内部。

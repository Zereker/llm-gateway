# 08. Observability

This document defines the observability contract for gateway logs, metrics, traces, Usage Events, and Content Log.

Observability data falls into three categories: runtime metrics for troubleshooting and scheduling feedback; Usage Events for billing facts; Content Log for audit and replay. The three are physically independent and are only correlated via `request_id` / `trace_id`.

## 1. Common Fields

All logs, spans, metric events, and async events across request paths should carry the following as much as possible:

| Field | Description |
|------|------|
| `request_id` | Gateway request ID; from header or generated by M1 |
| `trace_id` / `span_id` | tracing context |
| `account_id` | Primary account / billing subject |
| `sub_account_id` | Sub-account / operator |
| `api_key_id` | Credential ID; the raw API key is never logged |
| `service_id` | model service ID |
| `model` | Requested model |
| `routed_model` | Actual routed model; differs from `model` on cross-model fallback |
| `vendor` | endpoint vendor |
| `endpoint_id` | Actual endpoint |
| `protocol` | client protocol |
| `endpoint_protocol` | endpoint upstream protocol |
| `error_code` | Stable machine code |
| `error_class` | Behavioral classification |
| `attempt_role` | `primary` or `fallback`, used for cross-model fallback stats |

High-cardinality fields must not be casually turned into Prometheus labels. `request_id`, `trace_id`, and `api_key_id` by default only go into logs, traces, and async events — not into regular metric labels.

## 2. Logs

Logs use structured JSON. Code on the request path must call `slog.InfoContext` / `WarnContext` / `ErrorContext`; calling context-less `slog.Info` / `Error` / `Warn` directly is forbidden.

Recommended events:

| event | level | Description |
|-------|-------|------|
| `request.start` | debug | Request entering; disableable in production by default |
| `request.end` | info | Request completed; includes status, latency, usage summary |
| `auth.reject` | warn | API key / subscription / model service rejection |
| `ratelimit.reject` | warn | Client-side RPM/RPS rejection |
| `schedule.decision` | debug | Endpoint selection result and exclusion reason |
| `upstream.error` | warn/error | Upstream call failure |
| `usage.publish.error` | error | Usage event publish failure |
| `content_log.drop` | warn | Content log dropped due to backpressure |

Logs must not record API keys, decrypted endpoint secrets, or full request/response bodies. When content needs to be recorded, use the Content Log channel.

## 3. Metrics

Metric names use the `llm_gateway_` prefix. Histogram buckets are implementation-configured and should by default cover millisecond- to minute-scale requests.

| Metric | Type | Labels | Description |
|------|------|--------|------|
| `llm_gateway_http_requests_total` | counter | `method`, `route`, `status`, `error_class` | Total HTTP requests |
| `llm_gateway_http_request_duration_seconds` | histogram | `method`, `route`, `status`, `model`, `routed_model` | Gateway end-to-end latency |
| `llm_gateway_invoker_requests_total` | counter | `vendor`, `endpoint_id`, `model`, `endpoint_protocol`, `result`, `error_class` | Total upstream requests |
| `llm_gateway_invoker_duration_seconds` | histogram | `vendor`, `endpoint_id`, `model`, `result`, `error_class` | Upstream call latency |
| `llm_gateway_selector_attempts_total` | counter | `model`, `routed_model`, `vendor`, `endpoint_id`, `attempt_role`, `result`, `error_class` | endpoint attempt stats |
| `llm_gateway_selector_candidates` | histogram | `model`, `stage` | Candidate count; stage is list/eligible/cooldown/quota |
| `llm_gateway_scheduling_duration_seconds` | histogram | `model`, `attempts` | Total time for scheduling filter / pick / report |
| `llm_gateway_eligibility_duration_seconds` | histogram | `model` | Eligibility filtering time |
| `llm_gateway_ratelimit_decisions_total` | counter | `scope`, `dimension`, `result` | Client-side and endpoint-side rate limit decisions |
| `llm_gateway_ratelimit_charge_total` | counter | `dimension`, `result` | TPM post-charge result |
| `llm_gateway_ratelimit_fail_open_total` | counter | `scope`, `dimension` | Explicit fail-open occurrences |
| `llm_gateway_tpm_overflow_total` | counter | `layer`, `dimension` | Times the configured cap was exceeded after TPM post-charge |
| `llm_gateway_policy_cache_requests_total` | counter | `layer`, `result` | quota policy cache hit/miss/error |
| `llm_gateway_usage_tokens_total` | counter | `model`, `routed_model`, `vendor`, `direction` | token usage; `direction` is `input` / `output` (no `total`, to avoid doubling on sum); `source` / `confidence` are not metric labels — kept in the Usage Event and logs to control cardinality |
| `llm_gateway_usage_publish_total` | counter | `backend`, `result` | Usage Event publish result |
| `llm_gateway_content_log_publish_total` | counter | `backend`, `result`, `sampled` | Content Log publish result |
| `llm_gateway_outbox_buffer_size` | gauge | `backend` | Current async outbox buffer |
| `llm_gateway_outbox_publish_duration_seconds` | histogram | `driver`, `result` | outbox publish latency |
| `llm_gateway_outbox_dropped_total` | counter | `driver`, `reason` | Count of dropped outbox events |
| `llm_gateway_outbox_dlq_total` | counter | `driver`, `result` | DLQ write result |
| `llm_gateway_endpoint_misconfigured_total` | counter | `vendor`, `reason` | Startup-time endpoint config integrity check |
| `llm_gateway_request_aborted_by_shutdown_total` | counter | `route` | Requests interrupted by shutdown timeout |
| `llm_gateway_repo_cache_total` | counter | `table`, `result` | repo TTL LRU hit stats; `result` ∈ `hit\|miss\|error`; `table` ∈ `api_keys\|model_services\|endpoints_list\|endpoints_id\|quota_policies\|subscriptions` |
| `llm_gateway_repo_sql_load_total` | counter | `table`, `result` | repo direct SQL query stats; `result` ∈ `ok\|error\|not_found` |

repo cache metrics implementation: `pkg/repo/cache_metrics.go` (Metrics interface) + `cmd/gateway/repo_metrics.go` (Prom counter adapter).

When metrics are used for Runtime Scoring, the Scheduler does not read Prometheus directly; it should read the EMA / sliding-window summary in `EndpointStatsStore`. Metrics is the observability layer; `EndpointStatsStore` is internal scheduler state.

## 4. Tracing

OTel attributes prefer the `gen_ai.*` and HTTP semconv standards; when a standard field is missing, use
`llm_gateway.*`.

LLM-related attributes:

| Semantics | Attribute |
|------|-----------|
| operation | `gen_ai.operation.name` |
| Requested model | `gen_ai.request.model` |
| Actual routed model | `gen_ai.response.model` |
| vendor | `gen_ai.system` |
| input tokens | `gen_ai.usage.input_tokens` |
| output tokens | `gen_ai.usage.output_tokens` |
| total tokens | `gen_ai.usage.total_tokens` |
| TTFT (streaming) | `gen_ai.response.ttft_ms` |
| endpoint id | `llm_gateway.endpoint.id` |
| account id | `llm_gateway.account.id` |
| sub account id | `llm_gateway.sub_account.id` |
| api key id | `llm_gateway.api_key.id` |
| service id | `llm_gateway.service.id` |
| request id | `llm_gateway.request_id` |
| duration (ms) | `llm_gateway.duration_ms` |
| error code | `llm_gateway.error.code` |
| error class | `llm_gateway.error.class` |
| scheduler attempt | `llm_gateway.scheduler.attempt` |

HTTP semconv attributes (auto-written by the M1 root span, aligned with otelgin v0.68.0):

| Semantics | Attribute | Notes |
|------|-----------|------|
| HTTP method | `http.request.method` | `GET` / `POST` etc. |
| route template | `http.route` | `c.FullPath()`, not the actual URL, to avoid high cardinality |
| status code | `http.response.status_code` | Written by M1 after `c.Next()` |
| url scheme | `url.scheme` | `http` / `https` |
| client ip | `client.address` | gin `c.ClientIP()` |
| user-agent | `user_agent.original` | |

Span behavior:

- The M1 root span name is decided by `SpanNameFormatter(c)`, defaulting to `{METHOD} {route}` (consistent with
  otelgin); it falls back to `{METHOD}` when the route doesn't match. A custom function can be injected via
  `WithSpanNameFormatter` (e.g., to rewrite by modality).
- M1 root span kind = `SpanKindServer`; the propagator injected via `WithTraceContextPropagators`
  extracts the upstream traceparent.
- Error status: after `c.Next()`, M1 checks `rc.Error`; if non-empty, it calls `span.SetStatus(codes.Error, ...)`
  and writes `llm_gateway.error.{code,class}`; HTTP 5xx also sets Error; 4xx stays Unset
  (consistent with otelgin / HTTP semconv).
- `gin.Context.Errors` is synced to span events via `span.RecordError`.

Suggested span structure:

```text
gateway.request                     (M1, span name = "POST /v1/chat/completions")
  auth.lookup                        (M2)
  envelope.parse                     (M3)
  budget.check                       (M4, optional)
  catalog.resolve                    (M5)
  moderation.check                   (M8, optional)
  ratelimit.reserve                  (M6 pre-side)
  dispatch.request                   (M7, pkg/dispatch)
    dispatch.attempt                 (per attempt; attrs: model / endpoint / verdict)
      upstream.call                  (pkg/invoker)
      usage.extract                  (inside translator)
    dispatch.fallback (event)        (triggered by Switch action)
  ratelimit.charge_tpm               (M6 post-side, runs after c.Next())
  tracing.commit                     (M10)
    usage.publish
    content_log.publish
```

`dispatch.request` is the dispatcher's root span; attrs include `dispatch.model` /
`dispatch.group` / `dispatch.attempt_cap` / `dispatch.outcome` /
`dispatch.routed_model` / `dispatch.attempts` / `dispatch.http_code`.
Each attempt opens a `dispatch.attempt` child span, with attrs including `attempt.model` /
`attempt.index` / `attempt.candidates` / `attempt.eligible` /
`endpoint.id` / `endpoint.vendor` / `endpoint.protocol` /
`verdict.stage` / `verdict.class` / `verdict.http_code` / `verdict.reason`.
A fallback model switch is marked via a `dispatch.fallback` event on `dispatch.request`
(payload: `{from, to}`).

The spans listed above are a chronologically-ordered set of child spans under `gateway.request` (siblings); no nesting hierarchy is mandated — the implementation may decide which spans are parent/child based on actual scope. For example, if `usage.extract` is actually wrapped inside `upstream.call`, make it a child span there, but don't artificially nest spans just to "look tidy." `ratelimit.charge_tpm` / `usage.publish` happen after `c.Next()` (M6 post-side / M10) and should be direct children of `gateway.request`, not children of `upstream.call`.

In streaming responses, the `upstream.call` span covers from the first chunk to the last; TTFT is recorded as the `gen_ai.response.ttft_ms` attribute.

## 5. Usage Event

Usage Event is the billing-fact channel and does not carry large bodies. Default Kafka topic:

```text
billing.usage.recorded.v1
```

DLQ topic:

```text
billing.usage.recorded.v1.dlq
```

Topics are named by **domain.entity.event.version**, decoupled from the producing service name — downstream billing / reconciliation / quota consumers subscribe by business domain, and multiple services producing usage events still write to the same topic (see [05-metering-billing §5](./05-metering-billing.md#5-usage-outbox)).

The partition key is `account_id`; it falls back to `request_id` when missing. The payload uses a JSON envelope:

```json
{
  "schema_version": "usage.v1",
  "event_id": "01J...",
  "usage": {
    "input": 128,
    "output": 256,
    "total": 384,
    "raw": {},
    "source": "upstream",
    "estimator": "",
    "confidence": "exact",
    "meta": {
      "account_id":          "acct_abc",
      "sub_account_id":      "sub_001",
      "api_key_id":          "ak_xxx",
      "model":               "gpt-4o",
      "vendor":              "openai",
      "endpoint_id":         "12345",
      "service_id":          "svc_gpt4o",
      "model_service_id":    12345,
      "service_update_time": "2026-04-18T09:00:00Z",
      "request_id":          "req_...",
      "trace_id":            "4bf9...",
      "start_time":          "2026-05-16T09:59:57Z",
      "end_time":            "2026-05-16T10:00:00Z",
      "ttft_ms":             320,
      "total_latency":       2800
    }
  },
  "created_at": "2026-05-16T10:00:00Z"
}
```

`request_id` / `trace_id` only appear inside `usage.meta` — not duplicated at the envelope top level (to avoid dual-write inconsistency). `created_at` is the time the outbox enqueued the event; the actual request time is given by `usage.meta.start_time` / `usage.meta.end_time`. Breaking schema changes should preferably move to a new topic, e.g. `billing.usage.recorded.v2`, going through dual-write and a consumer cutover. The `.v1` in the topic name is topic-level physical isolation, a separate mechanism from the envelope's `schema_version`.

## 6. Content Log

Content Log records request/response content, which may contain PII, and must be kept separate from Usage Event. Disabled by default.

Only `none` / `file` output backends are supported. The gateway deliberately does **not** embed producers for Kafka / S3 / Loki etc. — Content Log is a logging/audit channel, and downstream often has multiple sinks (S3 archival + Loki search + Kafka content-safety post-review + training-data feedback); having the gateway handle multi-sink delivery itself would couple all those downstream availabilities to the main request path. The correct shape:

```text
gateway ──→ content.jsonl ──→ fluent-bit / vector ──┬─→ S3 / OSS
                                                    ├─→ Loki / ES
                                                    ├─→ Kafka topic (content safety post-review)
                                                    └─→ ...
```

File rotation / compression / cleanup is handled by external logrotate or a log collector (fluent-bit's tail input supports inode following), not inside the gateway process. Adding / adjusting sinks only requires changing the fluent-bit config, no gateway release needed. See [05-metering-billing §2](./05-metering-billing.md#2-content-log) for details.

The full driver and sampling / backpressure config schema is at [07-configuration §2 `content_log`](./07-configuration.md#2-gatewayyaml). This section only lists the fields meaningful for observability:

| Config | Default | Description |
|------|------|------|
| `driver` | `none` | `none` fully disables it; `file` writes local JSONL |
| `sample_rate` | `1.0` | Sampling ratio |
| `backpressure` | `drop_oldest` | Strategy when buffer is full; can be `drop_newest` / `block`; `block` requires `block_timeout` to be set |
| `buffer_size` | `1024` | Async queue size |

Content Log events must carry `request_id` / `trace_id` / `account_id` / `api_key_id` / `model` / `endpoint_id`, but are not required to be published in sync with Usage Event. Content Log failures do not affect the business response, nor do they affect Usage Event.

## 7. Error Response

All abort exits use a unified response:

```json
{
  "error": {
    "code": "ratelimit.exceeded",
    "message": "rate limit exceeded",
    "class": "capacity",
    "details": {},
    "request_id": "req_...",
    "trace_id": "4bf9..."
  }
}
```

`code` is the stable machine code; `class` is the internal behavioral classification; HTTP status is the protocol response and does not substitute for `class`. `details` may only contain safe diagnostic fields — no body, secrets, or raw sensitive upstream information.

## 8. Alerting Recommendations

Recommended first-version alerts:

- Sustained rise in `5xx` rate.
- Sustained rise in `upstream.error_class=transient|capacity`.
- Nonzero `usage.publish.result=failed`.
- Sustained rise in `content_log.drop`.
- Redis / DB health check failures.
- Scheduler eligible candidates at 0 for an extended period.

Alerts are based on metrics; troubleshooting jumps to trace/log; billing reconciliation uses Usage Event.
